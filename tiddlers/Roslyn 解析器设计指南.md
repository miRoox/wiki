本文档描述了 Roslyn 解析器的设计指南。这些并不是不能违反的硬性规则，请使用良好的判断力。在有理由且收益大于成本的情况下，可以偏离这些指南。

目前，解析器在很多地方即使没有充分理由，也不符合这些指南。这些指南是在解析器编写完成后才制定的。我们预计会在有机会时（尤其是当有具体收益时）对解析器进行重构，以使其符合这些指南。

为新功能设计语法模型（语法树的数模型）目前不在本文档的讨论范围内。

### **应该（DO）** 让解析器接受符合语法模型的输入文本

语法模型定义了语法分析与语义分析之间的契约。如果解析器能够为一段输入 token 序列构建出有意义的语法树，那么就应该交给语义分析阶段去报告该树在语义上是否有效。

解析器最重要的诊断信息是 **“缺少 token”** 和 **“意外的 token”**。

在某些情况下可能有理由违反这条规则。例如，语法模型中没有“优先级”的概念，但解析器会使用优先级来决定如何组装语法树。因此，优先级相关的错误会在解析器中报告。因为这种情况下语法树的形状会受到影响，我们不希望构建出非法的树，所以在此层级报告诊断是合理的。

然而，对于那些语法模型可以轻松适配，但规则本质上是受外部上下文驱动的**语义规则**，则应尽量避免在解析器中产生诊断。一个典型的例子是**语言版本（language version）检查**。除了少数例外情况（如 record 解析），解析本身实际上并不受语言版本影响。语言版本只是说明某个构造是否允许使用，而不是它是否能被成功识别并解析成语法模型。这些检查应该推迟到后续阶段进行，例如在声明表（decl-table）或绑定（binding）阶段进行（具体原因请参见本文档下方的 DOs/DON'Ts 部分）。  
注意：如果把这些检查留在解析器层级能显著避免在更高层级出现大量分散的检查（例如可能需要在几十个地方都增加检查），那么维护性成本的负面影响可能超过收益，这种情况下在解析器中保留检查也是可以接受的。

#### **不应该（DO NOT）** 使用环境上下文（ambient context）来指导解析器的行为

解析器应尽可能保持**上下文无关（context-free）**。仅在需要解决语言歧义时才使用上下文。例如，在既允许类型又允许表达式但优先解析为类型的上下文中（如 `is` 运算符的右侧），我们会将 `X.Y` 解析为类型。而在某些上下文中（例如 `case` 之后），只有当后面跟着标识符时才应解析为类型，此时我们会使用前瞻（look-ahead）来决定构建哪种语法树。

应尽量减少用于解析决策的上下文数量，因为这将显著提升**增量解析（incremental parsing）**的质量。

当然，也存在合理的违反理由。例如，语言规范明确要求对上下文敏感的情况：`await` 在其所在方法带有 `async` 修饰符时才被视为关键字。

### 不遵循这些 DOs/DON'Ts 的影响

上述 DO 和 DON'T 的目的并非追求语法或解析器设计的“纯粹性”，而是因为它们对编译器的**可维护性**、**正确性**，以及更高层级直接消费语法模型的组件（尤其是 IDE）的**性能**有重大影响。具体来说：

1. **环境上下文**的存在会严重影响增量解析的能力。环境上下文必须以某种方式被追踪，而在不同上下文之间进行增量复用可能会违反“增量重解析应产生与普通解析相同语法树”的不变性。这可能导致根本性的错误，只有重启宿主（host restart）才能解决。例如，当用户在方法上添加 `async` 修饰符时，这必然会影响该方法内部节点的复用，因为这些节点可能被单独解析。每一个环境上下文都可能产生这种影响，这极大增加了对增量编辑的推理难度，并可能在实际使用中导致难以诊断或复现的增量错误。

2. **除了“缺少 token”和“意外 token”之外的其他诊断**也会对增量解析产生影响。增量解析无法复用带有错误的节点，因为它无法判断错误的根本原因是否会被文件中其他位置的编辑修复。因此，任何额外诊断都会强制对该构造及其所有父节点进行完全重解析。这会增加增量解析的时间、导致更高的内存开销（因为需要创建新节点）。这些问题会直接影响 IDE 场景下的下游消费。在 IDE 中，**输入延迟（typing latency）** 是一个高优先级的 SLA 目标。许多功能必须在用户输入窗口内基于正确的语法树运行，例如括号自动插入（brace-insertion）和缩进。这些功能需要几乎瞬间完成，且都需要获取最新的 SyntaxTree。因此这些树是通过增量解析同步获取的，必须尽量减少额外工作，以便在可用时间片内完成所有任务。额外诊断会干扰这一点，增加不必要的 CPU 开销并造成内存抖动。

3. 许多 IDE 功能采用“不造成更多伤害”的设计理念。其核心做法是：如果语法树不是良好形成的（well-formed），则认为对其进行语法调整很可能会导致问题。这会影响自动格式化等功能——它们不想让糟糕的情况变得更糟。通过将诊断限制在“缺少 token”和“意外 token”上，更高层级的功能可以更容易地判断：“这个树是否坏到让我不应该继续操作？”  
   但如果树在语法模型上完全符合，却包含其他与语法形状无关的诊断（例如语言版本相关的语义问题），就会导致这些功能被不必要地禁用。让这些功能去判断具体的编译器错误码是非常脆弱的做法，会把它们绑定到不透明的诊断信息（位置、产生原因等），而不是可以简单查询的 API。这与增量解析面临的问题类似，只是发生在更高层级的组件上。

### 示例

以下是一些我们可能修改解析器以满足这些指南的例子，以及这样做能解决的问题：

1. **100l 警告：建议使用 `L` 表示 long 字面量**

   在解析器中产生这个警告看似是一件小事，但它会干扰增量解析。增量解析不会复用带有诊断的节点。因此，即使代码中存在这个有帮助的警告，也可能降低用户输入时的解析性能，导致编辑器响应变慢。

   将此警告移到语义分析阶段后，IDE 在用户输入时的响应性会更好。

2. **成员解析（Member parsing）**

   语法模型使用不同的节点分别表示构造函数和方法。当看到 `public M() {}` 这样的声明时，解析器会将函数名与外围类型的名称进行比较，以决定应该将其表示为构造函数，还是表示为一个缺少返回类型的方法。

   这里，外围类型的名称实际上是一种**环境上下文**，它影响了编译器的行为。

   如果我们在语法模型中将此类声明统一视为构造函数，而将名称比较放在语义分析阶段进行，那么就可以对外暴露一个**上下文无关**的公共 API 来解析成员声明。参见 GitHub 问题：<https://github.com/dotnet/roslyn/issues/367> —— 由于这个问题，我们一直无法实现这个功能请求。

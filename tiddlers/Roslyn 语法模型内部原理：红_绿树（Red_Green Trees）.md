本文档详细解释了 Roslyn 语法模型的内部架构，重点介绍支撑其高性能的“绿”层（green layer）。本文面向正在研究或使用 Roslyn 编译器内部实现的开发者。

有关公共 API 和语法树的一般使用方法，请参考官方文档：  
[使用 .NET Compiler Platform SDK 语法模型](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/work-with-syntax)。

本文假设您已熟悉上述内容，并深入探讨使语法模型高效运行的实现细节。

## 红/绿树模式（Red/Green Pattern）

Roslyn 的语法树采用了一种称为“红/绿树”的设计模式。其核心思想是维护同一语法结构的**两层并行表示**：

- **绿节点（Green nodes）**：内部的、不可变的表示形式。这些节点仅存储语法**种类（Kind）**、**宽度（Width，字符数）** 以及**子节点**。它们不存储绝对文本位置，也不存储父节点指针。
- **红节点（Red nodes）**：面向公众的包装器。这些节点提供熟悉的公共 API，包括 `Span`、`Parent` 以及强类型的子节点访问器。红节点是轻量级包装器，它将绿节点与位置和父级上下文相结合。

这种分离之所以存在，是因为**不可变性**与**高效导航**在本质上是矛盾的。一个不可变的节点无法存储父指针（因为同一个节点在不同上下文中可能有不同的父节点）。但用户需要父节点导航功能。红/绿模式很好地解决了这个问题：绿节点保持不可变且可共享，而红节点则按需计算位置和父节点，从而提供导航 API。

## 绿节点：核心设计

一个绿节点存储以下内容：

- **Kind**：该语法的类型（例如 `MethodDeclaration`、`IdentifierToken`）
- **Width**：该节点及其所有后代节点总共跨越的字符数
- **Children**：指向子绿节点的引用（通过槽位实现）
- **Diagnostics**：附加到该节点的任何语法错误

关键在于，绿节点**不存储**：

- **绝对位置**：绿节点不知道自己在源文件中的具体位置
- **父指针**：绿节点不知道自己被谁包含

这一设计选择是根本性的，其他一切优化都源于此。

### 基于槽位（Slot）的子节点访问

绿节点使用整数“槽位”（slots）来访问子节点。每种绿节点类型都知道自己有多少个槽位，以及每个槽位应存放何种子节点。例如，一个绿色的 `MethodDeclarationSyntax` 节点可能有以下槽位：属性（0）、修饰符（1）、返回类型（2）、标识符（3）、类型参数（4）、参数（5）、约束（6）、方法体（7）等等。

对于每种红语法节点类型，内部都对应有一个绿语法节点类型。因此，既有公开的红 `MethodDeclarationSyntax`，也有内部的绿 `MethodDeclarationSyntax`。绿类型可以通过槽位索引进行统一遍历，也可以通过与每个子节点对应的强类型属性进行访问。红层在通过公共 API 生成子节点时，会直接访问这些绿属性。

这种设计允许内部算法通过槽位索引对绿树进行泛型遍历（无需知道正在访问的具体节点类型），同时在已知节点类型时提供高效的强类型访问。

## 无位置、无父节点的优势

决定不在绿节点中存储位置和父节点，带来了多项强大的优化。

### 增量解析与子树复用

当用户编辑文件时，增量解析可以复用先前语法树的大部分内容。由于绿节点不编码绝对位置，一个原本位于位置 10,000 的方法声明，在前面发生编辑后可以直接复用到位置 10,050。绿节点本身完全相同；只有红包装器计算出的位置会发生变化。

这使得增量解析能在微秒级完成，对于典型编辑的内存复用率可接近 99.99%。详细信息请参阅 [[Roslyn 增量解析：深度剖析]]。

### 跨树共享

同一个绿节点可以出现在完全无关的语法树中。例如，空参数列表 `()` 对应的 `ParameterList` 节点。在一个大型解决方案中，这个结构可能出现在数千个方法中。由于绿节点没有位置或父节点信息，**同一个绿节点对象** 可以被所有这些地方共享。

这种共享通过绿节点缓存（下文介绍）自动实现，从而在解决方案级别节省大量内存。

### 树内共享：实际上是 DAG 而非树

同一个绿节点甚至可以在**单个**语法树中多次出现。如果一个文件中包含十个方法都使用空参数列表 `()`，这十个位置都可以指向同一个 `ParameterList` 绿节点。

这意味着“绿树”其实是个误称。绿结构实际上是一个**有向无环图（DAG）**，而非严格的树。多个父节点可以共享同一个子节点。这极大降低了具有重复构造的文件的内存占用。

红层会隐藏这种复杂性。每个红节点看起来都是一个具有唯一父节点和位置的独立对象，即使底层绿节点可能是共享的。

## 在绿层，一切都是节点

在公共红 API 中，语法模型对不同概念使用不同类型：

- `SyntaxNode`：类（引用类型），用于语法节点
- `SyntaxToken`：结构体（值类型），用于标记
- `SyntaxTrivia`：结构体，用于空白和注释
- `SyntaxList<T>`：结构体，用于节点列表

而在绿层，**所有这些都是堆分配的节点对象**。没有结构体。每个绿标记、每段绿 trivia、每个绿列表都是 `GreenNode` 的子类。

为什么？因为结构体无法参与上述的共享场景。如果标记在绿层是结构体，那么在多个位置复用同一个标识符时，就需要每次复制结构体。而使用绿节点对象时，我们可以直接指向同一个实例。

除了共享之外，这一选择还支持绿层的一个重要设计：**特化（specialization）**。绿树包含许多针对不同场景优化的特化实现（例如优化的标记表示和特化的列表实现）。这种特化对于实现良好的性能和内存占用至关重要，后文将详细讨论。

结构体不适合这个角色，因为它们无法参与继承层次结构，也无法支持在共同抽象背后有多种特化实现。通过将绿节点表示为对象，我们可以使用标准的继承技术自由地特化内部表示，同时在公共语法层暴露简单、轻量级的结构体 API。公共结构体实际上是这些底层绿节点的轻量级门面。

结果就是红包装器极其廉价。一个红 `SyntaxToken` 结构体仅仅包含：

- 一个指向父红节点的指针
- 一个整数位置
- 一个指向底层绿标记节点的指针

访问标记时**不会发生堆分配**。trivia 和列表在红层也是如此。

由于标记、列表和 trivia 通常占语法树元素的 **75% 或更多**，这种设计避免了遍历树时原本会产生的大量分配。

## 列表优化

列表在语法树中无处不在：参数列表、实参列表、语句列表、修饰符列表等等。Roslyn 在绿层对列表表示进行了大量优化。

### 空列表

空列表在绿层直接表示为 `null`，完全不分配内存。当被红 `SyntaxList<T>` 包装时，会检查 null 并返回 `Count = 0`。

### 单元素列表

恰好包含一个元素的列表不会分配绿列表节点。父绿节点会直接指向那个唯一的子元素。红 `SyntaxList<T>` 会检测到它指向的是一个非列表绿节点，从而返回 `Count = 1`，并提供对该元素的访问。

### 小型列表（2、3、4 个元素）

长度为 2、3、4 的列表有专用的绿节点子类（`WithTwoChildren`、`WithThreeChildren` 等）。这些子类将子指针直接存储在字段中，而不是使用数组。这样既避免了数组分配，也避免了访问数组元素的间接开销。

### 较大列表

包含 5 个或更多元素的列表则使用基于数组的表示。

#### 为什么这很重要

实际上，**90% 或更多的列表包含 4 个或更少的元素**。想想看：您写的方法中，包含 0、1、2、3 个参数的情况远多于 5 个以上参数的情况。实参列表、类型参数列表、特性列表等也都遵循类似的分布。

通过针对常见情况进行优化，Roslyn 为绝大多数列表避免了数组分配。红 `SyntaxList<T>` 和 `SeparatedSyntaxList<T>` 结构体透明地处理所有这些情况，无论底层表示如何，都提供统一的 API。

### 弱引用的红子节点

某些红列表可以使用弱引用（weak references）持有它们的红子节点。这允许当红树的部分不再被强引用时，被垃圾回收器（GC）回收，而不影响正确性。

这是安全的，因为只有当没有任何强引用持有某个红节点时，它才可能被回收。如果唯一的剩余引用是父列表持有的弱引用，那么就没有观察者能注意到该红节点消失。如果后续再次请求该子节点，会基于相同的底层绿节点创建一个新的红节点。由于红节点纯粹是不可变绿结构上的缓存，这种行为是不可观察的，在语义上等同于保留原始红节点。

#### 当前使用情况

目前，该优化仅应用于由成员或访问器拥有的块（`{ ... }`）（[参见此处](https://github.com/dotnet/roslyn/blob/744249e4f0afb645808e87aebec109a38d5dde8b/src/Compilers/CSharp/Portable/Syntax/CSharpSyntaxNode.cs#L518)）。这些块可能很大，且往往仅在分析期间临时需要。通过允许此类块的红列表弱引用其子节点，可以在不再需要时释放大量红内存，从而降低峰值内存占用，同时保留现有的红/绿树语义。

## 绿节点缓存

Roslyn 维护着一个常用绿节点的缓存。在解析过程中构造新绿节点时，解析器会先检查缓存中是否已存在等效节点。如果存在，则复用缓存中的节点，而不是分配新的。

由于组合爆炸问题，只有**子节点数量不超过 3 个**的节点才有资格被缓存。缓存本身可容纳 65,536 条目。尽管有这些限制，对 Roslyn 自身代码库进行解析的分析显示，**可缓存节点的命中率达到 55%**。这意味着超过一半的情况下，解析器可以复用现有分配，而不是创建新对象。

实现细节可参考 PR #80825，该 PR 简化并分析了 `SyntaxNodeCache`。

正是这种缓存机制实现了前文所述的跨树和树内共享。空参数列表 `()` 不仅在理论上可共享——它实际上被共享了，因为每次解析到该构造时，缓存都会返回同一个节点。

## 标记存储优化

由于绿节点是内部实现，我们可以以公开层面不方便的方式对其存储进行优化。标记就是一个很好的例子。

### 标识符标记（Identifier Tokens）

标识符标记存储指向其文本的指针（例如 `"myVariable"`）。由于文本已存储，宽度无需单独存储——可以直接从字符串长度计算得出。这为树中的每个标识符节省了空间。

### 关键字标记（Keyword Tokens）

关键字标记则相反。所有 `return` 关键字的文本完全相同，因此无需存储文本。关键字标记只需存储其 **Kind**，文本和宽度即可从 Kind 自动推断。例如，一个 `return` 关键字标记知道自己的 `SyntaxKind.ReturnKeyword`，文本 `"return"` 和宽度 `6` 自然得出。

### 预缓存的关键字变体

Roslyn 预先计算并缓存了所有不带 trivia 的关键字标记，以及带有常见前导和尾随空白模式的变体。例如 `void `（`void` 关键字后跟一个空格）。这个精确构造在源文件中频繁出现，每次出现都指向同一个字面绿节点实例。解析器无需分配任何东西——只需返回预缓存的节点即可。

这些优化对红 API 的用户是透明的，但它们显著降低了内存占用和解析过程中的分配压力。

### 预缓存的 Trivia

同样的缓存策略也应用于 trivia。常见模式（如单个空格、换行、典型的缩进序列——两个空格、四个空格、八个空格、制表符序列等）在文件内部和跨文件中会出现无数次。Roslyn 会预缓存这些常见模式，而不是每次都分配新的绿 trivia 节点。解决方案中每个单个空格 trivia 都指向同一个绿节点实例。

## 红节点创建：延迟且带缓存

红节点是在遍历语法树时按需创建的。如果您从未访问某个特定节点，其红包装器就永远不会被创建。

对于红 `SyntaxNode` 实例（类类型），一旦创建就需要缓存。这是因为节点具有引用标识——用户期望两次访问同一个子节点会返回同一个对象。缓存实现很简单：当红父节点被请求子节点时，它会检查该子红节点是否已创建。如果没有，则实例化它，并以原子方式写入其字段存储。哪个线程在竞争中获胜，就设置所有后续读取都会看到的指针。

对于红 `SyntaxToken`、`SyntaxTrivia` 和 `SyntaxList<T>`（均为结构体类型），则无需缓存。每次访问都会返回一个新的结构体实例，但由于结构体具有值语义，这是可以接受的。指向相同绿节点且位置相同的两个红结构体被认为是相等的。

### 对性能的影响

不需要遍历整棵树的特性只需为实际触及的部分付出代价。以 IDE 语法高亮为例：它只需检查屏幕上可见的标记。屏幕外方法的红节点永远不会被创建（或者如果之前创建过，在内存压力下可能被回收）。

结合增量解析带来的大量绿节点复用，这意味着在正常使用模式下，语法层分配的内存非常少：

- 绿节点在编辑间和文件间被大量复用
- 标记、列表和 trivia（占树 75%+）无需红分配
- 红节点分配被推迟到真正需要时
- 只有正在主动检查的树的部分会产生红节点开销

## 不可变性

绿节点是完全不可变的。一旦创建，绿节点永远不会改变。这带来了几个好处：

- **线程安全**：多个线程可以无锁地读取同一语法树
- **可预测性**：一旦持有某个节点的引用，它永远不会在您脚下发生变化
- **共享**：不可变对象可以自由共享，而无需防御性复制

当您需要“修改”语法树时（通过 `WithXxx` 方法或 `SyntaxFactory`），您会得到一个新树，它会尽可能多地与旧树共享结构。只有从根节点到修改点路径上的节点会被新分配，其余部分全部复用。

有关使用不可变语法树的更多信息，请参阅[官方文档](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/work-with-syntax)。

## 完整保真度（Full Fidelity）

Roslyn 语法树是**完整保真**的：源文件中的每个字符都在树中的某处有表示。空白、注释、预处理器指令，甚至格式错误的语法都会被保留。将树中所有标记和 trivia 按顺序拼接起来，可以精确重现原始源代码。

这一特性对工具场景至关重要。重构工具可以修改树的特定部分，同时保留用户在其他地方的格式。错误恢复机制确保部分编写的代码仍能产生可导航的树。

完整保真度还让增量解析能够精确判断编辑影响了树的哪些部分。详情请参阅 [[Roslyn 增量解析：深度剖析]]。

### 设计理念

代码库中存在大量针对不同节点配置、列表大小和标记类型的特化子类。这使得绿层能够进行深度优化。然而，红/绿分离的强大之处在于，所有这些复杂性都对用户隐藏了。公共 API 呈现的是一个干净、直观的语法树，支持父节点导航和绝对位置。用户无需了解绿节点、槽位访问、列表表示技巧或缓存策略。

但正因为这些内部优化，再结合上述使用模式，性能表现非常出色：

- 标记、列表和 trivia 无需红分配
- 红节点分配因延迟创建而极少
- 绿节点在树内和树间被大量共享
- 增量解析在编辑时复用几乎所有内容

复杂性被放在实现内部，从而让用户同时获得友好 API 和优秀性能。
